## 1. 概述

### 1.1 同步机制说明

本系统采用 **离线优先 (Offline-First)** 架构，支持 App 端在离线环境下采集数据，网络恢复后通过同步接口上传至服务端。

**核心特性：**
- **离线 UUID 映射**：每条离线记录生成唯一 `offlineUuid`，服务端通过 UUID 判断新增/更新
- **软删除标记**：通过 `offlineDeleted` 字段标记删除操作
- **闭环同步**：同步成功后立即重建用户专属 SQLite 并返回下载地址
- **事务性**：所有实体要么全部成功，要么全部回滚

### 1.2 同步流程总览

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    App 端    │    │    MinIO    │    │   服务端    │

       │                  │                  │

       │                  │                  │
       │ 2. 返回 minioId   │                  │

       │ 4. 解析 JSON 并落库                  │
       │ 5. 重建用户 SQLite                   │
       │ 6. 上传至 MinIO    │                  │
```

### 2.2 核心设计模式

**1. UUID 映射机制**

```java
// 服务端维护 offlineUuid → serverId 映射
Map<String, Long> uuidMap = new HashMap<>();

// 处理 BiObject 时，先解析 parentUuid 获取 parentId
Long parentId = uuidMap.get(item.getParentUuid());
item.setParentId(parentId);
```

**2. 多轮处理（BiObject 树形结构）**

```java
// 最多 5 轮，确保父子关系正确建立
for (int i = 0; i < 5; i++) {
    int roundCount = 0;
    for (BiObject item : list) {
        // 跳过已处理
        if (processedUuids.contains(item.getOfflineUuid())) continue;
        
        // 检查父节点是否已解析
        Long parentId = uuidMap.get(item.getParentUuid());
        if (parentId == null) continue; // 父节点未就绪，跳过
        
        // 处理当前节点
        process(item);
        roundCount++;
    }
    if (processedUuids.size() == total || roundCount == 0) break;
}
```

**3. 事务边界**

```java
@Override
@Transactional(rollbackFor = Exception.class)
public SyncResultVo syncUpload(Map<String, Object> dataMap) {
    // 所有实体处理在同一事务中
    processBuildings(...);
    processBiObjects(...);
    // ...任何异常都会回滚
}
```

## 5. 数据模型

### 5.1 通用字段规范

所有参与同步的表必须包含：

| 字段 | 类型 | 说明 | 默认值 |
|------|------|------|--------|
| `offline_uuid` | TEXT | 离线 UUID（唯一索引） | - |
| `is_offline_data` | INTEGER | 0=云端数据，1=离线生成 | 0 |
| `offline_deleted` | INTEGER | 0=正常，1=已删除 | 0 |

### 5.2 外键 UUID 字段

| 表 | UUID 字段 | 关联目标 |
|----|----------|---------|
| `bi_building` | `root_object_uuid` | → `bi_object.offline_uuid` |
| `bi_object` | `parent_uuid` | → `bi_object.offline_uuid` (自引用) |
| `bi_object` | `building_uuid` | → `bi_building.offline_uuid` |
| `bi_component` | `object_uuid` | → `bi_object.offline_uuid` |
| `bi_disease` | `building_uuid` | → `bi_building.offline_uuid` |
| `bi_disease` | `object_uuid` | → `bi_object.offline_uuid` |
| `bi_disease` | `component_uuid` | → `bi_component.offline_uuid` |
| `bi_disease_detail` | `disease_uuid` | → `bi_disease.offline_uuid` |
| `bi_attachment` | `offline_subject_uuid` | → 对应主体的 `offline_uuid` |

### 5.3 实体处理顺序

```
1. Building      (无依赖)
2. BiObject      (依赖 Building、BiObject 父节点)
3. Component     (依赖 BiObject)
4. Disease       (依赖 Building、BiObject、Component)
5. DiseaseDetail (依赖 Disease)
6. Attachment    (依赖任意主体)
7. BiObjectComponent (依赖 BiObject、Component)
```

#### 用例 6：并发同步（事务隔离）

```java
@Test
@DisplayName("E2E-006: 并发同步 - 事务隔离测试")
public void testSyncUpload_Concurrent_Isolation() throws Exception {
    // Given: 两个线程同时提交不同的数据
    CountDownLatch latch = new CountDownLatch(2);
    List<SyncResultVo> results = Collections.synchronizedList(new ArrayList<>());
    
    Runnable task1 = () -> {
        try {
            Map<String, Object> payload1 = buildPayload("thread-1-uuid");
            SyncResultVo result = syncUploadService.syncUpload(payload1);
            results.add(result);
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            latch.countDown();
        }
    };
    
    Runnable task2 = () -> {
        try {
            Map<String, Object> payload2 = buildPayload("thread-2-uuid");
            SyncResultVo result = syncUploadService.syncUpload(payload2);
            results.add(result);
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
```

## 8. 常见问题

### Q1: 同步时 UUID 重复如何处理？

**答：** 当前采用覆盖策略，后提交的数据会覆盖已有记录。如需避免，可在插入前检查 `is_offline_data` 状态。

### Q2: BiObject 处理为什么要多轮循环？

**答：** 因为 App 端可能乱序提交（子节点在前），通过多轮循环确保父节点先被解析。最多 5 轮，若仍有未处理记录则放弃。

### Q3: 事务回滚条件是什么？

**答：** 任何未捕获的异常都会触发 `@Transactional` 回滚。当前实体会记录错误但继续处理下一个，不会整体回滚。

### Q4: SQLite 生成失败如何排查？

**答：** 
1. 检查 `bi_project`、`bi_task`、`bi_building` 关联关系
2. 查看 `logs/sys-info.log` 中的详细错误
3. 验证 MinIO 连接和 bucket 权限

### Q5: 多设备同步同一用户数据会怎样？

**答：** 当前版本不支持并发同步。后提交的会覆盖先提交的数据。建议实现设备锁或会话管理。

### Q6: 附件删除如何处理？

**答：** 当前版本附件仅支持预上传，删除逻辑暂不处理。建议在业务层实现软删除标记。

### Q7: 如何清理过期的 ID 映射？

**答：** 定期清理 `bi_id_mapping` 表中超过 90 天的记录：

```sql
DELETE FROM bi_id_mapping 
WHERE create_time < DATE_SUB(NOW(), INTERVAL 90 DAY);
```
